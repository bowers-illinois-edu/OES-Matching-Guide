The OES Guide to Making and Evaluating Matched Designs

Approach 1: Problem-First Structure (Practitioner-Focused)

1. When Should You Use Matching?

1.1 Decision Framework

  • Flowchart: Treatment assignment → Sample characteristics → Research question → Method recommendation
  • Key decision points:
    • Is your treatment binary, multi-level, or continuous?
    • Do you have a clear comparison group?
    • Are you seeking causal estimates or descriptive comparisons?
    • What are your data constraints (sample size, missingness)?

1.2 Contexts Where Matching Excels

  • Observational studies with selection into treatment
  • When you have rich covariate information
  • Policy evaluations without experimental assignment
  • When you need transparent, interpretable comparisons
  • Studies where exact balance on key variables is critical

1.3 When NOT to Use Matching

  • True experiments (matching is redundant)
  • Weak overlap between treated and control groups
  • Limited covariate information
  • When regression discontinuity or IV designs are available
  • Very small sample sizes (case-by-case)

1.4 Alternatives to Consider

  • Regression adjustment (and why it’s different)
  • Inverse probability weighting
  • Regression discontinuity designs
  • Instrumental variables
  • Difference-in-differences
  • Synthetic control methods

1.5 Common Pitfalls and Red Flags

  • “Fishing” for balance by trying many matching approaches
  • Matching on post-treatment variables
  • Over-matching (matching away treatment effects)
  • Ignoring common support problems
  • Confusing balance with bias elimination

2.4 What Randomization Doesn’t Do

  • Doesn’t guarantee exact balance
  • Doesn’t solve non-compliance
  • Doesn’t address spillovers or interference
  • Limited to observed confounders in practice

2.5 What Matching Can Achieve

  • Approximating experimental comparison groups
  • Creating defensible counterfactuals
  • Making assumptions explicit and testable
  • Reducing model dependence

2.6 What Matching Cannot Achieve

  • Critical: Cannot control for unobserved confounders
  • Cannot fix fundamental lack of overlap
  • Cannot create data where none exists
  • Not a substitute for good research design

2.7 Core Assumptions for Causal Interpretation

  • Ignorability/No unmeasured confounding: Treatment assignment independent of potential outcomes given covariates
  • Overlap/Common support: Positive probability of treatment for all covariate values
  • SUTVA: Stable unit treatment value assumption
  • Making these assumptions plausible through design

3. The Matching Toolkit: Methods Overview

3.1 Quick Reference Table

Method Treatment Type Set Structure Software Best For
Pair Matching Binary Fixed pairs optmatch, designmatch Maximum transparency
Full Matching Binary Variable sets optmatch Maximum sample use
Bipartite (designmatch) Binary Fixed k:1 designmatch Fine-tuned constraints
Non-bipartite Continuous/Multi-level Variable sets designmatch Dose-response

3.2 Bipartite vs. Non-bipartite Matching

  • Bipartite: Separate treated and control groups
    • Used when treatment is binary
    • Creates strata comparing treated to control units
    • Most common in practice
  • Non-bipartite: No clear treated/control division
    • Used with continuous treatments or doses
    • Creates strata varying in treatment level
    • Each stratum has range of treatment values

3.3 Pair Matching vs. Full Matching

  • Pair matching: One treated, one control per set
    • Advantages: Maximum simplicity, easy interpretation
    • Disadvantages: Discards data, may force poor matches
    • Effective sample size considerations
  • Full matching: Variable number of treated and controls
    • Advantages: Uses all data, optimizes overall balance
    • Disadvantages: More complex sets to interpret
    • When to prefer each approach

3.4 Distance Metrics Compared

  • Mahalanobis distance: Weights all covariates equally (after scaling)
    • When to use: Theory suggests all covariates matter equally
    • Rank-based version for robustness
  • Propensity score distance: Weights by predictiveness
    • When to use: Some covariates strongly predict treatment
    • Requires careful model specification
  • Absolute/Euclidean distance: Single variable differences
    • When to use: One variable dominates concern
  • Combined distances: Adding multiple distance matrices
    • When to use: Multiple dimensions of similarity matter

3.5 Software Ecosystem

  • optmatch: Network flow optimization, fast, flexible
  • designmatch: Mixed integer programming, more constraints
  • Other options: MatchIt, Matching, CBPS (brief mentions)
  • When to use which package

  • match_on(): Creating distance matrices

5.3 Key Parameters and Their Effects

5.3.1 pairmatch() Parameters

pairmatch(
  distance,           # Distance matrix
  controls = 1,       # Number of controls per treated
  data,              # Data frame
  remove.unmatchables # Keep or drop unmatched units
)
  • controls: Fixed ratio (e.g., 2 means 1:2 matching)
  • remove.unmatchables: TRUE drops units that can’t be matched
  • Trade-offs of each parameter choice

5.3.2 fullmatch() Parameters

fullmatch(
  distance,
  min.controls,     # Minimum controls per treated
  max.controls,     # Maximum controls per treated
  data,
  tol              # Optimization tolerance
)
  • min/max.controls: Constrains set sizes
    • Larger range = better balance, more complex sets
    • Smaller range = simpler interpretation, may sacrifice balance
  • tol: Smaller values = better solution, slower computation
  • Effective sample size implications

6. Method Details: Bipartite Matching with designmatch

6.1 When to Use designmatch

  • Need more fine-grained control over constraints
  • Want to specify “near” constraints (penalties, not hard limits)
  • Working with integer programming optimization
  • Specific balance requirements

6.2 Core Function: bmatch()

6.3 Key Parameters and Their Effects

6.3.1 Basic bmatch() Syntax

bmatch(
  t_ind,           # Treatment indicator (sorted!)
  dist_mat,        # Distance matrix
  n_controls,      # Fixed number of controls
  total_groups,    # Number of matched sets
  solver           # Optimization solver settings
)
  • Critical: Data must be sorted by treatment (decreasing)
  • n_controls: Fixed ratio only (no variable sets like fullmatch)
  • total_groups: Determines how many units are matched

6.3.2 Constraint Parameters

bmatch(
  ...,
  subset_weight,   # Weight on minimizing distance vs. maximizing matches
  near_exact,      # Variables to nearly match
  exact,           # Variables to exactly match
  near_pairs       # Pairs of variables to be near each other
)
  • subset_weight: 0 = minimize distance only; 1 = maximize matches
  • near_exact: Soft constraint (penalty for deviation)
  • exact: Hard constraint (must match exactly)
  • near_pairs: Keep specified variables close together

6.3.3 Solver Options

solver = list(
  name = "highs",
  approximate = 1,  # Use heuristics (faster)
  t_max = 1000,    # Time limit in seconds
  trace = 1        # Verbosity
)
  • Solver choice: “highs”, “gurobi”, “symphony”
  • Trade-offs: speed vs. optimality

6.4 Comparison to optmatch

Feature optmatch designmatch
Algorithm Network flow Mixed integer programming
Variable set sizes Yes (fullmatch) No
Near constraints No Yes
Speed Faster Slower
Flexibility High Very high

9.2.2 Weighting Schemes

Block-Size Weights (Horvitz-Thompson Style)

difference_in_means(outcome ~ treatment, blocks = matched_sets, data)
  • Weights by number of treated in each set
  • Unbiased under randomization within sets
  • ATT-style estimator

Precision Weights (Fixed Effects Style)

lm_robust(outcome ~ treatment, fixed_effects = ~matched_sets, data)
# OR equivalently:
lm_robust(outcome ~ treatment + matched_sets, data)
  • Harmonic mean weights
  • Lower variance estimates
  • May be biased but often preferred in practice

9.2.3 Which Weights to Use?
